# 실전 교훈 — 채팅 시스템 개발에서 배운 것

버전: 1.0 (2026-04-29 작성)
관련 문서: `debugging_guide.md` (키오스크 디버깅 원칙)

## 0. 이 문서의 목적

이 문서는 **다음 프로젝트에서 같은 실수를 반복하지 않기 위한** 실전 사례 모음이다.
키오스크 프로젝트의 `debugging_guide.md`가 "원칙"이라면, 이 문서는 "경험"이다.

각 항목은 다음 형식으로 적는다.
- 증상: 무엇이 보였는가
- 진짜 원인: 실제로 무슨 일이 일어났는가
- 해결: 어떻게 바로잡았는가
- 교훈: 다음에 어떻게 예방할 것인가

## 1. 환경 변수 / 비밀 관리

### 1-1. dotenv가 환경 변수를 안 덮어쓰는 문제

**증상**: `.env`에 `ANTHROPIC_API_KEY`를 넣었는데 `process.env.ANTHROPIC_API_KEY`가 `undefined`. 그런데 `dotenv.config().parsed`에는 키가 보임.

**진짜 원인**: 부모 프로세스(Claude Code 등 일부 도구)가 동일 이름의 환경 변수를 **빈 문자열**로 미리 점유하고 있었다. dotenv는 기본 동작상 기존 env를 덮어쓰지 않으므로, 빈 값이 `.env` 값을 차단했다.

**해결**: `require('dotenv').config({ override: true })`

**교훈**:
- 환경 변수가 안 들어올 땐 `.env` 파일 문제만 의심하지 말고, **부모 셸의 env 충돌**도 확인하라
- 같은 이름의 환경 변수가 여러 출처에서 올 수 있다는 가능성을 항상 염두에 둠
- 의심스러우면 `node -e "console.log(process.env.NAME)"`을 dotenv 없이 실행해 시스템에 이미 그 변수가 있는지 확인

### 1-2. API 키가 잘려서 인증 실패

**증상**: API 호출이 250ms만에 401 인증 오류로 실패.

**진짜 원인**: 사용자가 콘솔에서 키를 복사할 때 끝부분이 잘려서 전달됨. 키 길이가 비정상적으로 짧았음(94자 vs 정상 108자).

**교훈**:
- API 키처럼 긴 비밀 값은 **길이를 먼저 검증**한다 (`process.env.KEY?.length`)
- 250ms 내 4xx 응답은 보통 **인증/검증 실패**, 그 이상이면 **백엔드 로직 실패**라는 직관을 가져라
- 키를 받을 때 "이게 끝까지 복사된 거 맞나"를 한 번 더 확인

## 2. Git / 워크플로우

### 2-1. 임베디드 git 저장소를 일반 add 시도

**증상**: `git add yakorea-questions/` 했더니 "embedded git repository" 경고가 뜨고, 실제 파일이 아닌 **submodule pointer 한 줄**만 커밋됨.

**진짜 원인**: `yakorea-questions/` 디렉토리에 자체 `.git`이 있는 별도 저장소였음. 메인 저장소에서는 그 디렉토리를 통째로 submodule처럼 취급함.

**해결**: 그 저장소로 들어가서 (`cd yakorea-questions`) 직접 add/commit/push.

**교훈**:
- `find . -name .git -type d` 같은 명령으로 프로젝트 안 모든 git 저장소를 미리 파악
- 별도 저장소면 그 저장소의 origin/branch도 따로 관리
- "왜 push했는데 배포가 안 되지?" 싶을 때 첫 번째 의심 후보

### 2-2. 브랜치 vs 직접 main 커밋의 기준

**상황**: 어떤 작업은 브랜치를 만들어 PR로 머지하고, 어떤 작업은 main에 바로 커밋. 기준이 모호.

**정한 규칙**:
- **바로 main**: CSS/문구/시각 변경, 1-2 파일, 한 줄로 되돌릴 수 있는 변경, 서버 재시작·테스트 불필요
- **브랜치 생성**: 새 기능, 서버 로직 변경, 여러 파일 동시 수정, 통합 테스트 필요, 잘못되면 운영에 직접 영향

**교훈**: 한 줄 요약 — **"운영 코드 건드리면 브랜치, 눈에만 보이는 거 바꾸면 바로 커밋"**

### 2-3. 배경 프로세스가 살아 있는 채로 재시작 시도

**증상**: `npm start`했는데 `EADDRINUSE: 3001` 에러.

**진짜 원인**: 이전 서버가 백그라운드에서 살아있음. 명시적으로 종료하지 않음.

**해결**: `netstat -ano | grep :3001`로 PID 찾아 `Stop-Process -Id N -Force`.

**교훈**:
- 백그라운드 작업은 **시작 시 task ID 기록 → 작업 끝나면 명시적 종료**
- 재시작 전에 "이전 인스턴스가 살아있는지" 먼저 확인
- 자동 재시작이 필요하면 `nodemon` 같은 도구를 처음부터 셋업

## 3. 인프라 / 배포

### 3-1. 같은 코드를 두 곳에 배포 중인지 모름

**증상**: Railway에 `worthy-kindness`와 `fabulous-stillness` 두 프로젝트가 같은 GitHub 저장소를 보고 있음. 둘 다 동시에 배포되고 있어서 어떤 게 진짜 운영 중인지 모름.

**해결**: 실제 손님이 사용하는 위젯의 `<script src>` URL을 확인 → `projectclaude-production-5351`이 운영 중. 다른 하나 삭제.

**교훈**:
- "이게 진짜 운영 중인 거 맞나"를 **실제 트래픽 진입점**(웹페이지 소스, DNS, CDN 설정)에서 역추적
- 인프라 리소스에는 작업 중간에 만들어진 **유령 자원**이 종종 남는다. 정기 점검 필요
- 클라우드 콘솔에서 보이는 "활성" 상태와 실제 사용 여부는 다를 수 있다

### 3-2. 로컬 file://에서 http://localhost 스크립트 차단

**증상**: `file:///C:/test.html`을 열었는데 `<script src="http://localhost:3001/widget.js">`가 로드 안 됨.

**진짜 원인**: 브라우저 보안 정책 — `file://`와 `http://`는 다른 origin으로 간주, 혼합 콘텐츠 차단.

**해결**: 테스트 페이지를 서버 정적 디렉토리에 두고 `http://localhost:3001/test.html`로 접근.

**교훈**:
- 브라우저에서 테스트할 땐 항상 **같은 origin에서 접근**하도록 셋업
- "왜 콘솔에 에러도 없는데 스크립트가 실행 안 되지?" → 브라우저 콘솔에서 차단된 리소스가 있는지 Network 탭 확인

## 4. CSS / 스타일

### 4-1. !important 우선순위 충돌

**증상**: `.category-heading { font-size: 0.85em }` 적용했는데 화면에서는 여전히 큼.

**진짜 원인**: 같은 프로젝트의 다른 CSS 파일 `common.css`에서 `h2, .headline { font-size: 40px !important }` 가 더 강한 우선순위로 덮고 있었음.

**해결**: 더 구체적 선택자(`.category-heading`)에 같은 `!important`로 덮어쓰기.

**교훈**:
- 스타일이 안 먹을 땐 **DevTools → Computed**로 실제 적용된 값과 그 출처를 본다
- 같은 셀렉터에 `!important`가 여러 개일 땐 **선택자 구체성**(specificity)이 결정자
- `!important` 남발한 코드를 만나면 그 자리에서 같은 무기로 덮어야 함

## 5. 외부 API 통합

### 5-1. 통합 테스트 실패의 원인 분리

**상황**: Phase 5 통합 테스트 6/8 통과 → 코드가 잘못됐나? 분석해보니 **코드는 정상**, API 잔액 부족(400)이나 인증 실패(401)가 원인.

**교훈**:
- 통합 테스트 실패 시 **HTTP 상태 코드로 1차 분류**:
  - 400/401/403 → 외부 의존성(인증, 권한, 잔액, 입력 형식)
  - 5xx → 외부 서비스 문제
  - 타임아웃 → 네트워크 또는 외부 서비스 응답 지연
  - 200인데 결과 이상 → 비로소 우리 코드 의심
- 테스트 코드에 외부 의존성 상태(API 키 존재, 크레딧 보유 등)를 **사전 체크하는 헬스 체크 단계**를 두면 좋음

### 5-2. 비용 통제 안전장치

LLM 폴백 같은 종량제 외부 API를 통합할 땐 시작부터 다음을 박는다:

- **토글 플래그** (DB 한 줄로 즉시 OFF)
- **레이트 제한** (룸당 N회, 분당 M회)
- **MAX_TOKENS 캡** (하나의 호출 비용 상한)
- **호출 로그 테이블** (input/output/cache_read 토큰, latency 모두 기록)
- **Graceful fallback** (API 다운 시 기존 흐름으로 자동 복귀)
- **기본값 OFF로 머지** → 모니터링 후 수동 ON

이 다섯 개를 **기능 첫 커밋에 같이 박는다**. "나중에 추가" 하면 안 됨.

## 6. 작업 진행 방식

### 6-1. 사용자에게 묻지 말고 직접 할 수 있는 것은 직접 한다

이번 작업 중에 "Supabase에서 이 SQL 한 줄 실행하세요"라고 사용자에게 시킨 적이 있는데, 사실 서버가 같은 Supabase에 접속해 있으므로 Node.js 한 줄로 직접 실행 가능했음.

**교훈**:
- 사용자에게 액션을 떠넘기기 전에 "내가 가진 도구로 직접 할 수 있나?"를 먼저 확인
- 비밀번호·API 키 발급·결제처럼 **사용자만 할 수 있는 일**과 **나도 할 수 있는 일**을 구분

### 6-2. 컨텍스트 압축 후 작업 재개

긴 세션이 컨텍스트 한계에 가까워지면 자동 압축이 일어나고, 압축 후에는 이전 대화 요약본만 남는다. 이 시점에서:

- 이전에 시작한 백그라운드 프로세스의 task ID는 새 세션에서 무효화될 수 있음
- 이전에 만든 git 워크트리/브랜치 상태를 다시 확인해야 함
- 요약 시점에서 끊긴 작업의 "어디까지 갔는지"를 명확히 파악

**교훈**:
- 긴 작업은 **중간에 커밋을 자주 남겨** 압축이 일어나도 git 상태로 복원 가능하게
- 핵심 결정 사항은 **decisions.md** 같은 파일에 기록 (압축 후에도 살아남음)

## 7. 최종 체크리스트 — 다음 프로젝트 시작할 때

- [ ] `.env.example` 작성 + `.env`는 `.gitignore`
- [ ] `dotenv.config({ override: true })` 사용 여부 결정 (개발 환경에서 충돌 가능성 검토)
- [ ] 임베디드 git 저장소가 있다면 처음부터 별도 관리 방안 정하기 (submodule? 분리?)
- [ ] 인프라 리소스 (Railway·Vercel·Supabase 등) **운영 vs 테스트 명확히 라벨링**
- [ ] 외부 유료 API 사용 시 토글·레이트제한·로깅·MAX_TOKENS 5종 세트 박기
- [ ] 백그라운드 프로세스 시작/종료 절차 문서화
- [ ] 큰 변경은 브랜치, 작은 변경은 main 직접 — 기준 미리 합의
- [ ] 시각적 변경 후엔 새로고침으로 즉시 검증, 로직 변경 후엔 통합 테스트로 검증
