# 야코리아 키오스크 (셀프 체크인 웹)

카운터에서 손님이 직접 체크인/안내를 받을 수 있는 HTML 기반 키오스크 웹입니다.  
목표는 “직원 개입 최소화 + 외국인 손님도 혼자서 이해 가능한 안내”입니다.

이 문서는 두 가지를 동시에 합니다.
1) 운영/설계의 기준(원칙)  
2) 유지보수 방법(변경 절차)

※ “지원 언어/폴더 구조/파일 목록”은 시간이 지나며 바뀔 수 있습니다.  
바뀔 수 있는 항목은 ‘현황’으로 기록하고, 변경 시에는 반드시 ‘변경 절차’를 따른 뒤 이 문서를 업데이트합니다.

---

## 0) 이 문서를 어떻게 업데이트할까

- 문서가 “틀리게” 되는 걸 막으려면, 아래 규칙을 지킵니다.
  - 바뀌면 안 되는 것은 `1) 핵심 원칙`에만 적습니다.
  - 바뀔 수 있는 것은 `2) 현황(스냅샷)`에 적습니다.
  - 바꾸는 방법은 `3) 변경 절차`에 적습니다.
  - 실제로 바꿨으면 `4) 변경 기록`에 1줄 남깁니다.

---

## 1) 핵심 원칙 (바뀌면 안 되는 기준)

### 1-1. 운영 정책 핵심
- 체크인 시작: **15:00(오후 3시) 이후**
- 야간: **프론트 데스크 CLOSED 안내**
- 청소 시간: **청소 시간 안내**
- 체크아웃 안내: **체크아웃 관련 안내 제공(운영 방식에 맞게)**

※ 운영 정책이 바뀌면 “시간대별 라우팅”과 “문구(i18n JSON)”가 같이 바뀌어야 합니다.

### 1-2. 안정성 우선 원칙
- 이미 잘 작동하는 기능은 임의로 삭제/변경하지 않습니다.
- 필요 여부가 불확실하면 먼저 운영자(나)에게 확인한 뒤 적용합니다.
- 변경은 “작게, 자주” 하되, 매번 실행/검증 가능한 단위로 합니다.

### 1-3. 보안 원칙
- 비밀번호/출입정보 같은 민감정보를 외부에 노출시키지 않습니다.
- 저장소 공개/공유 전에는 민감정보가 코드/데이터에 남아 있는지 점검합니다.
- 장기적으로는 하드코딩을 줄이고 “분리 저장”을 지향합니다.

### 1-4. UI/UX 디자인 원칙 (터치 인터페이스/가이드라인)

키오스크는 일반 웹보다 **버튼을 더 크게 설계해야 합니다.**  
이유는 손가락 터치 정확도 때문입니다.

#### 권장 크기

| 요소 | 권장값 |
|---|---|
| 버튼 높이 | 70px ~ 100px |
| 버튼 가로 | 180px ~ 260px |
| 버튼 비율 | 2.5:1 ~ 3:1 |

**예시 규격:**
- 높이(height): 80px
- 너비(width): 200px
- Minimum touch target: 70px
- Recommended touch target: 80px+

#### 버튼 스타일 가이드 (CSS 예시)
키오스크 버튼 작성 시 참고할 기본 스타일 마크다운입니다. 터치감을 위해 둥근 모서리와 굵은 폰트를 사용하며, 반응형(Hover/Touch) 피드백 크기를 키웁니다.

```css
.kiosk-button {
  width: 100%;       /* 컨테이너에 맞추거나 권장 가로 폭(ex: 200px) 고정 */
  height: 100%;      /* 컨테이너에 맞추거나 권장 세로 높이(ex: 80px) 고정 */
  min-height: 80px;  /* 터치 타겟 최소 높이 보장 */
  
  font-size: 2.2em;
  font-weight: bold;
  
  background-color: white;
  color: #333;
  border: 4px solid #333;
  border-radius: 30px; /* 둥근 모서리로 부드러운 터치감 유도 */
  
  cursor: pointer;
  transition: all 0.3s;
  
  display: flex;
  align-items: center;
  justify-content: center;
}

/* 터치/클릭 시각적 피드백 */
.kiosk-button:hover,
.kiosk-button:active {
  transform: scale(1.05);
  border-color: #007bff;
  color: #007bff;
}
```

## 2) 현황 (현재 스냅샷, 바뀔 수 있음)

### 2-1. 화면 흐름 (User Flow)
- 시작: `index.html` (최상위 루트)
- 언어 선택(현재 지원 언어는 아래 참고)
- 현재 시간에 따라 다음 페이지로 이동
  - 15:00~23:00: `pages/self_check_in.html`
  - 23:00~05:00: `pages/reception_closed.html`
  - 05:00~11:00: `pages/no_check_out.html`
  - 그 외: `pages/cleaning_time.html`
- 체크인 흐름(주요)
  - `pages/self_check_in.html` → “다음” → `pages/reservation_number.html`
  - `pages/reservation_number.html`
    - 이름 또는 예약번호 입력
    - `data/reservations.xlsx`에서 예약 검색
    - 결과에 따라 안내(방/사물함 비번, 동선, 영상 등)

### 2-2. 지원 언어 (현재)
- 현재: ko / en / zh / ja
- 언어는 추가될 수 있음(ru, th, hi 등). 추가 방법은 `3-1. 언어 추가 절차` 참고.

### 2-3. 다국어(i18n) 구조
- 페이지별 JSON에서 문구 로드
  - `data/self_check_in.json`
  - `data/reservation_number.json`
  - `data/reception_closed.json`
  - `data/no_check_out.json`
  - `data/cleaning_time.json`
  - `data/QnA.json`
- 언어는 URL 파라미터로 결정
  - 예: `pages/reservation_number.html?lang=ko`

### 2-4. 예약 데이터(엑셀) 로딩 방식
- 파일: `data/reservations.xlsx`
- 로딩/파싱 코드: `js/reservation_number.js`
- 주의:
  - 엑셀 컬럼 위치가 바뀌면 검색이 깨질 수 있음
  - 운영에서 가장 중요한 건 “엑셀 최신화 자동화/반자동화”임

### 2-5. 민감정보 위치(현재)
- 객실 비밀번호/사물함 비밀번호가 코드에 존재할 수 있음 (`js/reservation_number.js`)
- 공유/공개 전에 반드시 점검 필요
- 개선 방향은 `3-4. 민감정보 관리 절차` 참고

### 2-6. 폴더 구조 (현재)
- `index.html` : 시작 페이지 (언어 선택 및 라우팅)
- `start_kiosk.bat` : 키오스크 모드 자동 실행 배치 스크립트
- `pages/` : 세부 화면 HTML
- `js/` : 페이지별 로직 및 공통 로직 (`app.js`, `i18n.js`, `config.js` 등)
- `css/` : 스타일
- `data/` : 번역 JSON + 예약 엑셀 + 운영 데이터
- `audio/` : 음성 안내 파일
- `images/` : 안내 이미지

※ 폴더/파일은 늘어나거나 이름이 바뀔 수 있습니다. 변경 시 기준은 `3-2. 구조 변경 절차`를 따릅니다.

### 2-7. 로컬 실행 방법
- 브라우저 보안 정책 때문에 로컬 서버로 실행해야 합니다.
  - 가장 쉬운 방법: **`start_kiosk.bat`** 더블클릭 (파이썬 서버와 Chrome 키오스크 모드가 자동 실행됨)
  - 수동 실행 시:
    - 터미널에서 `python -m http.server 8000` 입력
    - 접속: `http://localhost:8000/index.html`

## 3) 변경 절차 (바뀔 때 이렇게 한다)

### 3-1. 언어 추가 절차 (예: 러시아어 ru 추가)

1) 언어 코드 결정  
- 예: ru, th, hi 등 (2글자 권장)

2) i18n JSON에 새 언어 키 추가  
- 모든 JSON 파일에 같은 언어 키를 추가해야 “어떤 페이지는 번역 있고, 어떤 페이지는 없는” 상태를 막을 수 있음
- 최소 요구:
  - `data/self_check_in.json`
  - `data/reservation_number.json`
  - `data/reception_closed.json`
  - `data/no_check_out.json`
  - `data/cleaning_time.json`
  - `data/QnA.json`

3) 언어 선택 UI에 버튼 추가  
- `index.html`에서 언어 선택용 HTML 버튼 추가
- 각 언어별 `?lang=ru` 쿼리 파라미터가 URL에 포함되도록 구성 (필요 시 `js/app.js` 등 공통 함수 참고)

4) 폰트/글자 깨짐 점검  
- 키릴 문자, 태국어, 인도계 문자는 폰트/줄바꿈 문제가 생길 수 있음
- 실제 키오스크 화면(해상도)에서 확인

5) README 업데이트  
- `2-2. 지원 언어(현재)`에 ru 추가
- `4) 변경 기록`에 1줄 추가

### 3-2. 폴더 구조/파일 이동 절차

폴더 구조는 바뀔 수 있지만, 기준을 지켜야 유지보수가 쉬워집니다.

기준:
- 페이지(HTML)는 `pages/` (단, 진입점은 `index.html`)
- 화면 로직(JS)은 `js/`
- 번역/데이터(JSON/XLSX)는 `data/`
- 리소스(이미지/오디오)는 각각 `images/`, `audio/`

절차:
1) 이동/추가 이유를 한 줄로 적는다(이 문서의 변경 기록에 남김)
2) 경로 참조가 깨지는 곳을 전부 수정한다(HTML의 script/link, JS의 fetch 경로)
3) 로컬 서버에서 전체 흐름을 최소 1회 끝까지 실행한다
4) README의 `2-6. 폴더 구조(현재)` 업데이트 + `4) 변경 기록` 추가

### 3-3. 운영 정책(시간대/체크인 조건) 변경 절차

1) 정책을 문장으로 먼저 확정한다(예: 체크인 가능 시간, 야간 안내 시간 등)
2) `index.html`의 하단 스크립트 내 시간대 분기(라우팅) 로직 수정
3) 관련 페이지의 i18n 문구(JSON) 수정
4) 실제 키오스크에서 “경계 시간” 테스트
- 예: 14:59, 15:00, 22:59, 23:00, 04:59, 05:00 등
5) README의 `1-1. 운영 정책 핵심`과 `4) 변경 기록` 업데이트

### 3-4. 민감정보(비밀번호 등) 관리 절차

원칙:
- 외부 공유 전에 “민감정보가 코드/데이터에 남아 있지 않은지” 반드시 확인

권장 개선 방향(선호 순)
- (A) 민감정보를 별도 파일로 분리하고, 로컬에서만 관리(저장소 제외)
- (B) 더 안전하게는 서버/백엔드에서 내려주기(권한 체크 포함)

점검 체크리스트:
- `js/` 안에 비밀번호 하드코딩 여부 (`js/reservation_number.js` 주의)
- `data/`에 민감정보 들어간 파일 존재 여부
- 저장소 공개/공유 전, 민감정보 제거/치환 여부
