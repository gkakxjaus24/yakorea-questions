# 키오스크 프로젝트 전용 디버깅 운영 문서
버전: 요약본
목적: 키오스크 코드가 원하는 대로 작동하지 않을 때, 감으로 수정하지 않고 원인을 빠르게 찾고 최소 수정으로 해결하기 위한 기준 문서

## 1. 이 문서의 목적

이 문서는 내 키오스크 프로젝트를 디버깅할 때 반드시 따라야 하는 공통 기준이다.  
목표는 세 가지다.

첫째, 문제가 생겼을 때 원인도 모른 채 이곳저곳을 동시에 수정하지 않도록 한다.  
둘째, 현재 정상 작동 중인 기능을 최대한 유지하면서 문제 구간만 정확히 고친다.  
셋째, AI가 코드를 수정할 때도 내 의도와 프로젝트 구조를 해치지 않도록 작업 범위를 통제한다.

이 프로젝트는 일반 웹페이지가 아니라 실제 게스트하우스 현장에서 외국인 손님이 직접 사용하는 키오스크다.  
따라서 단순히 “코드가 돌아간다”만으로는 부족하다.  
터치스크린에서 잘 눌려야 하고, 언어 전환이 자연스러워야 하며, 음성 안내가 실제 기기에서 재생되어야 하고, 노트북과 키오스크 PC에서 모두 일관되게 동작해야 한다.

즉, 이 프로젝트의 디버깅은 개발자용 테스트가 아니라 실제 손님 사용 환경 기준으로 판단해야 한다.

## 2. 가장 중요한 원칙

### 원칙 1. 디버깅은 수정이 아니라 원인 찾기다
문제가 생기면 바로 코드를 고치지 않는다.  
먼저 어디까지는 정상이고, 어디서부터 비정상인지 확인한다.

### 원칙 2. 한 번에 하나만 바꾼다
HTML, CSS, JavaScript, 언어 데이터, 음성 경로를 한 번에 같이 바꾸지 않는다.  
한 번에 한 부분만 수정하고 바로 테스트한다.

### 원칙 3. 로그 없이 추측하지 않는다
“아마 이 부분인 것 같다”는 식으로 접근하지 않는다.  
반드시 로그, 콘솔 에러, 네트워크 응답, 실제 입력값으로 확인한다.

### 원칙 4. 현재 작동 중인 기능은 보존한다
이미 잘 되는 언어 선택, 버튼 이동, 예약 검증, 음성 재생 같은 기능은 확실한 이유 없이는 구조를 바꾸지 않는다.  
필요한 부분만 최소 수정한다.

### 원칙 5. 노트북 테스트만으로 끝내지 않는다
내 노트북에서 잘 된다고 끝난 것이 아니다.  
키오스크 PC, 터치 환경, 실제 배포 환경에서 다시 확인해야 한다.

## 3. 이 프로젝트에서 자주 문제 되는 핵심 영역

이 키오스크 프로젝트는 특히 아래 다섯 영역에서 문제가 자주 발생할 수 있다.

### 1) 버튼 및 화면 전환
예:
- 다음 버튼을 눌렀는데 아무 반응이 없음
- 이전 화면으로 안 돌아감
- 특정 단계에서 흐름이 멈춤

### 2) 다국어 전환
예:
- 한국어는 되는데 중국어 문구만 비어 있음
- 일본어 버튼을 눌렀는데 일부 문장이 영어로 남아 있음
- 번역 키 이름이 맞지 않아 문구가 출력되지 않음

### 3) 예약 정보 검증
예:
- 이름 또는 예약번호 입력 후 검증이 안 됨
- 특정 데이터 형식에서만 실패함
- 공백, 대소문자, 날짜 형식 때문에 비교가 틀어짐

### 4) 음성 및 미디어 재생
예:
- 노트북에서는 음성이 나오는데 키오스크에서는 안 나옴
- 버튼을 눌러도 음성 파일이 재생되지 않음
- 오디오 경로, 권한, 자동재생 정책 문제

### 5) 환경 차이
예:
- 로컬에서는 되는데 배포 후 안 됨
- 크롬 버전 차이
- 터치 이벤트와 클릭 이벤트 차이
- 캐시 때문에 예전 코드가 계속 실행됨

## 4. 문제 발생 시 기본 디버깅 절차

문제가 생기면 아래 순서를 반드시 따른다.

### 1단계. 기대 동작을 한 줄로 적는다
예:
- 중국어 버튼을 누르면 모든 문구가 중국어로 바뀌어야 한다.
- 예약번호 입력 후 다음 버튼을 누르면 검증 후 다음 화면으로 넘어가야 한다.
- 음성 버튼을 누르면 해당 언어 음성이 즉시 1회 재생되어야 한다.

### 2단계. 실제 동작을 한 줄로 적는다
예:
- 버튼을 눌러도 아무 반응이 없다.
- 일부 문구만 번역되고 나머지는 그대로다.
- 노트북에서는 음성이 나오지만 키오스크에서는 나오지 않는다.

### 3단계. 재현 절차를 적는다
예:
1. 메인 화면 진입
2. 중국어 선택
3. 이름 입력
4. 예약번호 입력
5. 다음 버튼 클릭
6. 화면 전환 안 됨

재현 절차가 있어야 문제를 다시 같은 방식으로 확인할 수 있다.

### 4단계. 최근 수정한 부분부터 확인한다
버그는 대개 최근 바뀐 코드 근처에서 생긴다.  
먼저 아래를 확인한다.

- 최근 바뀐 HTML 구조
- 버튼 id 또는 class 이름
- 이벤트 연결 방식
- 다국어 JSON 키
- 오디오 파일명 또는 경로
- fetch 또는 비동기 처리
- CSS display, visibility, z-index
- 배포 경로와 로컬 경로 차이

### 5단계. 로그를 함수 시작, 중간, 끝에 넣는다
반드시 흐름을 끊어서 본다.  
예를 들어 버튼 문제라면 다음 순서로 확인한다.

- 버튼 클릭이 실제로 발생하는가
- 함수가 호출되는가
- 조건문을 통과하는가
- API 요청 또는 데이터 검증까지 가는가
- 마지막 화면 전환 코드까지 도달하는가

핵심은 “어디서 끊기는지”를 찾는 것이다.

### 6단계. 단순 로그가 아니라 값도 같이 확인한다
예:
- 현재 언어값
- 입력된 이름
- 예약번호
- 응답 데이터
- 오디오 경로
- 선택된 DOM 요소

실제로는 함수가 실행됐어도 값이 틀려서 실패하는 경우가 매우 많다.

## 5. 키오스크 프로젝트에서 특히 자주 놓치는 진짜 원인

이 프로젝트는 겉으로 보이는 증상과 실제 원인이 다른 경우가 많다.

### 1) 선택자 오타
JavaScript에서 찾는 id와 HTML의 실제 id가 다르면 버튼이 안 먹는다.

### 2) 다국어 키 불일치
한국어에는 키가 있는데 중국어나 일본어 번역 키가 빠지면 일부 문장이 비어 보일 수 있다.

### 3) 공백 또는 데이터 형식 문제
예약번호 앞뒤 공백, 문자열과 숫자 비교, 날짜 형식 차이 때문에 검증이 실패할 수 있다.

### 4) CSS 때문에 안 보이는 문제
실제로는 다음 단계로 넘어갔는데 display, opacity, z-index 때문에 화면이 안 바뀐 것처럼 보일 수 있다.

### 5) 오디오 정책 또는 기기 설정 문제
키오스크 PC에서는 브라우저 정책, 볼륨 설정, 출력 장치 문제 때문에 음성이 안 나올 수 있다.  
이 경우 코드만 의심하면 안 된다.

### 6) 캐시 문제
수정한 최신 코드가 아니라 예전 JS나 CSS가 계속 실행되는 경우가 있다.  
강력 새로고침, 캐시 제거 여부를 꼭 확인한다.

## 6. 노트북에서는 되는데 키오스크에서는 안 될 때 체크할 것

이 경우 가장 먼저 환경 차이를 의심한다.

- 브라우저 종류와 버전이 같은가
- 배포 파일이 실제로 최신 버전인가
- 캐시가 남아 있지 않은가
- 터치 환경에서 click만 듣고 있지는 않은가
- 음성 출력 장치와 볼륨 설정이 맞는가
- 오디오 파일 경로가 키오스크 환경에서도 유효한가
- 화면 해상도 차이 때문에 요소가 가려지지는 않는가
- 인터넷 또는 네트워크 요청이 막히지는 않는가

즉, “같은 코드인데 기기마다 다르게 보이는 문제”는 코드 버그와 환경 버그를 분리해서 봐야 한다.

## 7. AI가 이 프로젝트를 디버깅할 때 지켜야 할 방식

AI는 아래 규칙을 반드시 따른다.

첫째, 먼저 기대 동작과 실제 동작을 정리한다.  
둘째, 원인 후보를 우선순위로 제시한다.  
셋째, 관련 없는 파일이나 구조는 건드리지 않는다.  
넷째, 원인을 모르면 먼저 로그 위치를 제안한다.  
다섯째, 수정 코드를 줄 때는 수정 범위를 명확히 말한다.  
여섯째, 기존 정상 기능을 보존하는 최소 수정 방식으로 제안한다.  
일곱째, 확실하지 않으면 확실하지 않다고 표시한다.

이 프로젝트는 내가 직접 유지보수하는 구조이므로, AI가 보기 좋게 구조 전체를 갈아엎는 방식은 금지한다.

## 8. 디버깅 요청할 때 포함해야 하는 정보

AI에게 문제를 맡길 때는 아래 항목을 같이 준다.

- 기대 동작
- 실제 동작
- 재현 절차
- 콘솔 에러
- 최근 변경 사항
- 관련 코드 구간
- 노트북과 키오스크 중 어디서 발생하는지
- 어떤 언어에서만 발생하는지
- 어떤 버튼 또는 어떤 단계에서 멈추는지

좋은 디버깅은 “왜 안 되지?”라고 묻는 것이 아니라, “어디까지는 되고 어디서부터 깨지는지”를 자료와 함께 주는 것이다.

## 9. 최종 원칙

이 프로젝트 디버깅의 핵심은 아래 한 문장이다.

문제를 감으로 고치지 말고, 흐름을 잘라서 어디까지 정상인지 확인한 뒤, 최소 수정으로 해결한다.

이 문서를 항상 우선 기준으로 삼는다.  
특히 다국어, 예약 검증, 화면 전환, 음성 재생, 키오스크 환경 차이는 이 프로젝트의 핵심 위험 구간이므로, 수정 전후 테스트를 반드시 수행한다.
